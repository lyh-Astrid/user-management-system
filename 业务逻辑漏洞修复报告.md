## 1. 业务逻辑漏洞概述

### 1.1 什么是业务逻辑漏洞

业务逻辑漏洞（Business Logic Vulnerability）是指应用程序在实现业务功能时，由于**对业务流程的假设不完整或边界条件处理不当**，导致攻击者可以通过非预期的操作顺序、参数值或用户角色来获取利益或破坏系统。

### 1.2 与传统安全漏洞的区别

```
传统安全漏洞（如SQL注入）:
  攻击者绕过系统防护，执行非授权的代码

业务逻辑漏洞:
  攻击者在系统授权范围内，利用业务规则的漏洞
  进行非预期的操作 —— "合法地做坏事"
```

## 2. 漏洞详情与复现

### 2.1 BIZ-01：充值金额未校验正负

#### 2.1.1 漏洞信息

| 项目 | 内容 |
|------|------|
| **漏洞编号** | BIZ-01 |
| **位置** | `app.py` 第 289-299 行（修复前） |
| **风险等级** | 🔴 严重 |
| **漏洞类型** | 业务逻辑 - 参数校验缺失 |

#### 2.1.2 漏洞代码（修复前）

```python
@app.route("/recharge", methods=["POST"])
def recharge():
    """充值接口（不校验 amount 正负）"""
    user_id = request.form.get("user_id", type=int)
    amount = request.form.get("amount", type=float)

    if user_id not in USERS_BY_ID:
        return redirect("/")

    # ❌ 直接做加法，没有校验 amount 是否为负数
    USERS_BY_ID[user_id]["balance"] += amount
```

#### 2.1.3 攻击复现

```
攻击目标: 通过传递负数金额，将任意用户的余额扣减至零或负数

请求:
  POST /recharge
  user_id = 1
  amount = -99999

服务器执行:
  USERS_BY_ID[1]["balance"] += (-99999)
  # admin余额: 99999 - 99999 = 0

请求:
  POST /recharge
  user_id = 1
  amount = -100000

服务器执行:
  USERS_BY_ID[1]["balance"] += (-100000)
  # admin余额: 0 - 100000 = -100000
```

**漏洞效果**：
- ✅ 攻击者可清空任意用户的余额
- ✅ 攻击者可让用户余额变为负数
- ✅ 无需任何权限校验

### 2.2 BIZ-02：充值接口未做登录校验

#### 2.2.1 漏洞信息

| 项目 | 内容 |
|------|------|
| **漏洞编号** | BIZ-02 |
| **位置** | `app.py` 第 286-300 行（修复前） |
| **风险等级** | 🔴 严重 |
| **漏洞类型** | 认证缺失 |

#### 2.2.2 漏洞代码

```python
@app.route("/recharge", methods=["POST"])
def recharge():
    """充值接口（不校验 amount 正负）"""
    user_id = request.form.get("user_id", type=int)
    amount = request.form.get("amount", type=float)

    # ❌ 没有检查用户是否已登录
    if user_id not in USERS_BY_ID:
        return redirect("/")

    USERS_BY_ID[user_id]["balance"] += amount
```

#### 2.2.3 攻击复现

```bash
# 攻击者无需登录，直接发送充值请求
# 只要知道 user_id，即可修改任意用户余额

$ curl -X POST http://localhost:5000/recharge \
  -d "user_id=1&amount=1000"

$ curl -X POST http://localhost:5000/recharge \
  -d "user_id=1&amount=-50000"
```

**漏洞效果**：未登录用户可以通过直接 POST 请求来修改任意用户的余额。

### 2.3 BIZ-03：个人中心越权访问（IDOR）

#### 2.3.1 漏洞信息

| 项目 | 内容 |
|------|------|
| **漏洞编号** | BIZ-03 |
| **位置** | `app.py` 第 272-283 行（修复前） |
| **风险等级** | 🔴 严重 |
| **漏洞类型** | 不安全的直接对象引用（IDOR） |

#### 2.3.2 漏洞代码

```python
@app.route("/profile", methods=["GET"])
def profile():
    """个人中心（通过 URL 参数 user_id 查询任意用户资料）"""
    user_id = request.args.get("user_id", type=int)

    # ❌ 没有判断当前登录用户
    # ❌ 没有阻止访问其他用户的资料
    user_data = USERS_BY_ID.get(user_id) if user_id else None
```

#### 2.3.3 攻击复现

```bash
# 未登录状态下，通过枚举 user_id 查看任意用户资料
$ curl http://localhost:5000/profile?user_id=1
# 返回 admin 的邮箱、手机、余额

$ curl http://localhost:5000/profile?user_id=2
# 返回 alice 的邮箱、手机、余额

$ curl http://localhost:5000/profile?user_id=3
# 返回已注册的其他用户的资料
```

**漏洞效果**：任何未登录用户都可以通过修改 URL 参数 `user_id` 查看系统中任意用户的个人资料（邮箱、手机号、余额等敏感信息）。

### 2.4 BIZ-04：余额可为负数

#### 2.4.1 漏洞信息

| 项目 | 内容 |
|------|------|
| **漏洞编号** | BIZ-04 |
| **位置** | `app.py` 第 295-297 行（修复前） |
| **风险等级** | 🟡 中危 |
| **漏洞类型** | 业务规则违反 |

#### 2.4.2 漏洞分析

充值功能的业务逻辑是"增加余额"，但系统允许通过负数充值来"扣减余额"，且没有设置余额下限。攻击者可以将任意用户的余额扣减到负值，破坏系统数据完整性。

```python
# ❌ 没有检查余额下限
# ❌ 没有拦截负数充值
USERS_BY_ID[user_id]["balance"] += amount
```

## 3. 修复方案详解

### 3.1 修复一：充值金额正数校验

```python
# ❌ 修复前
amount = request.form.get("amount", type=float)
USERS_BY_ID[user_id]["balance"] += amount    # 直接累加，未校验

# ✅ 修复后
amount = request.form.get("amount", type=float)

# 金额必须为正数
if not amount or amount <= 0:
    return redirect(f"/profile")

USERS_BY_ID[user_id]["balance"] += amount
```

**修复效果**：
| 充值金额 | 修复前 | 修复后 |
|:--------:|:------:|:------:|
| `amount=100` | ✅ 充值成功 | ✅ 充值成功 |
| `amount=0` | ✅ 余额不变 | ❌ 拦截，重定向回个人中心 |
| `amount=-500` | ✅ 余额扣减 | ❌ 拦截，重定向回个人中心 |
| `amount=-99999` | ✅ 余额清空 | ❌ 拦截，重定向回个人中心 |

### 3.2 修复二：充值接口登录认证

```python
# ❌ 修复前
@app.route("/recharge", methods=["POST"])
def recharge():
    user_id = request.form.get("user_id", type=int)
    amount = request.form.get("amount", type=float)
    # ... 没有登录检查直接处理充值

# ✅ 修复后
@app.route("/recharge", methods=["POST"])
def recharge():
    username = session.get("username")
    if not username:          # ✅ 检查登录状态
        return redirect("/login")

    current_user = USERS.get(username)
    if not current_user:
        return redirect("/login")

    user_id = current_user["id"]   # ✅ 从 session 获取，不从表单获取
    amount = request.form.get("amount", type=float)
```

**修复效果**：
| 请求来源 | 修复前 | 修复后 |
|:---------|:------:|:------:|
| 已登录用户 | ✅ 可充值 | ✅ 可充值 |
| 未登录用户 | ✅ 可充值（漏洞） | ❌ 重定向到 `/login` |

### 3.3 修复三：个人中心权限控制

```python
# ❌ 修复前
@app.route("/profile", methods=["GET"])
def profile():
    user_id = request.args.get("user_id", type=int)  # 从URL获取
    user_data = USERS_BY_ID.get(user_id)
    return render_template("profile.html", user=display_data)

# ✅ 修复后
@app.route("/profile", methods=["GET"])
def profile():
    username = session.get("username")
    if not username:          # ✅ 检查登录状态
        return redirect("/login")

    current_user = USERS.get(username)
    if not current_user:
        return redirect("/login")

    user_id = current_user["id"]     # ✅ 从 session 获取
    user_data = USERS_BY_ID.get(user_id)
    return render_template("profile.html", user=display_data)
```

**修复效果**：
| 请求方式 | 修复前 | 修复后 |
|:---------|:------:|:------:|
| 已登录用户访问 `/profile` | ✅ 显示本人资料 | ✅ 显示本人资料 |
| 未登录用户访问 `/profile` | ✅ 显示任意用户资料（漏洞） | ❌ 重定向到 `/login` |
| `?user_id=2` URL 参数 | ✅ 可越权查看 | ❌ 参数被忽略，使用 session |

## 5. 安全加固总结

### 5.1 修复前后架构对比

```
❌ 修复前（漏洞架构）:

  /profile?user_id=X  ──→  任意用户资料（无需登录）
  /recharge           ──→  任意操作（无需登录，金额不限）
  /login              ──→  明文泄露管理员密码


✅ 修复后（安全架构）:

  /profile            ──→  Session → 仅本人资料（需登录）
  /recharge           ──→  Session + amount>0 → 本人充值（需登录）
  /login              ──→  无敏感信息泄露
```

### 5.2 业务逻辑安全设计原则

```
┌──────────────────────────────────────────────────────────┐
│              🥇 业务逻辑安全设计原则                       │
│                                                          │
│   1. 认证先行：所有敏感操作必须先验证用户身份                │
│   2. 权限绑定：用户只能操作自己的数据（user_id从session取）│
│   3. 参数校验：所有用户输入必须校验格式和范围                │
│   4. 最小信息：不在前端暴露不应公开的敏感信息                │
│   5. 正向逻辑：只允许正向操作（充值只能加，不能减）          │
│   6. 服务端信任：永远不信任客户端传递的 user_id 等关键参数   │
│                                                          │
│     "Never trust the client. Never trust user input."      │
│      永远不信任客户端。永远不信任用户输入。                  │
└──────────────────────────────────────────────────────────┘
```

### 5.3 各安全层修复对照

| 安全层级 | 修复内容 | 对应漏洞 |
|:--------:|:---------|:--------:|
| **认证层** | `recharge()` 添加 `session.get("username")` 检查 | BIZ-02 |
| **认证层** | `profile()` 添加 `session.get("username")` 检查 | BIZ-03 |
| **授权层** | `user_id` 从 `session` 获取，不从 URL/表单获取 | BIZ-03 |
| **校验层** | `amount <= 0` 拦截非正数 | BIZ-01, BIZ-04 |
| **信息隐藏** | 删除 login.html 中泄露密码的 HTML 注释 | BIZ-05 |
