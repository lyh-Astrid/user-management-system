## 1. 漏洞概述

### 1.1 什么是文件包含漏洞

文件包含漏洞（File Inclusion Vulnerability）是指 Web 应用程序在动态加载文件时，**未对用户传入的文件路径参数进行充分校验**，导致攻击者可以读取服务器上的任意文件。当攻击者通过 `../` 序列跳出受限目录时，也称为**路径遍历漏洞（Path Traversal）**。

### 1.2 文件包含漏洞的危害

| 危害类型 | 示例 | 严重程度 |
|:---------|:-----|:--------:|
| 🔴 **系统密码泄露** | 读取 `/etc/shadow`，获取密码哈希 | **灾难性** |
| 🔴 **应用源码泄露** | 读取 `app.py`，发现更多漏洞 | **严重** |
| 🔴 **数据库文件窃取** | 读取 `users.db`，获取所有用户数据 | **灾难性** |
| 🔴 **服务器配置泄露** | 读取 `/etc/nginx/nginx.conf` | **严重** |
| 🔴 **远程代码执行（RFI）** | 从远程服务器包含恶意文件 | **灾难性** |
| 🟡 **源代码审计** | 读取业务逻辑代码，寻找弱点 | **中危** |

### 1.3 本次发现漏洞总览

| 编号 | 攻击方式 | 读取目标 | 风险等级 | 状态 |
|:----:|:---------|:---------|:--------:|:----:|

## 2. 文件包含漏洞原理

### 2.1 路径遍历的攻击原理

文件包含漏洞的本质是**应用程序将用户输入直接拼接到文件路径中，攻击者利用 `../` (dot-dot-slash) 序列跳出受限目录**。

```
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  服务器文件系统结构:                                           │
│                                                              │
│  /opt/                                                       │
│  ├── pages/            ← 期望的文件访问目录                      │
│  │   └── help.html                                           │
│  ├── app.py                                                  │
│  └── data/                                                   │
│      └── users.db                                            │
│                                                              │
│  /etc/                                                       │
│  ├── passwd            ← 攻击目标                              │
│  └── shadow                                                  │
│                                                              │
│  正常请求:                                                    │
│    name = "help"                                              │
│    path = pages/ + "help"          → pages/help.html    ✅     │
│                                                              │
│  攻击请求:                                                    │
│    name = "../../../etc/passwd"                               │
│    path = pages/ + "../../../etc/passwd"                      │
│         → pages/../../../etc/passwd                           │
│         → /etc/passwd               ← 目录跳转成功！  ❌      │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 2.2 漏洞代码分析

**漏洞代码（修复前）**：

```python
@app.route("/page", methods=["GET"])
def page():
    name = request.args.get("name", "")

    # ❌ 直接拼接用户输入到路径中
    filepath = os.path.join(PAGE_DIR, name)

    # ❌ 直接读取文件，无任何安全校验
    if os.path.isfile(filepath):
        with open(filepath, "r", encoding="utf-8") as f:
            page_content = f.read()
    else:
        # ❌ 尝试加 .html 后缀后再次读取
        filepath_html = filepath + ".html"
        if os.path.isfile(filepath_html):
            with open(filepath_html, "r", encoding="utf-8") as f:
                page_content = f.read()
```

**漏洞根本原因**（从技术层面分析）：

| 问题 | 说明 |
|:----|:------|
| **① 缺乏白名单** | 没有限定可访问的页面名称，任何字符串都可作为文件名 |
| **② 路径未归一化** | 没有调用 `os.path.realpath()` 解析 `../` 序列 |
| **③ 缺乏目录边界检查** | 没有验证最终路径是否在 `pages/` 目录内 |
| **④ 无文件类型限制** | 任何后缀的文件都可读取（.py, .db, .conf 等） |
| **⑤ 异常处理不足** | 对敏感文件读取尝试没有日志记录 |

```bash
# → 返回"页面不存在"

# 危险输入

```

## 4. 攻击原理深度分析

### 4.1 `os.path.join` 与路径遍历的关系

`os.path.join()` 在处理路径时，如果后续参数是**绝对路径**（以 `/` 开头），则会**丢弃前面所有的路径部分**：

```python
os.path.join("/opt/pages", "../../etc/passwd")
# → "/opt/pages/../../etc/passwd"
# → "/etc/passwd"（归一化后）

os.path.join("/opt/pages", "/etc/passwd")
# → "/etc/passwd"（❌ 绝对路径，前面全部丢弃！）
```

如果攻击者直接传入 `name=/etc/passwd`（以 `/` 开头），`os.path.join` 会直接丢弃 `/opt/pages/` 前缀，使目录边界完全失效。

### 4.2 攻击链组合

```
信息收集阶段:
    读取 /etc/passwd → 获取系统用户名
    读取 app.py → 获取 secret_key、数据库路径
    读取 users.db → 获取所有用户数据

深入攻击阶段:
    secret_key → Session 伪造 → 管理员权限登录
    数据库信息 → SQL 注入增强攻击
    系统用户名 → SSH 爆破

完全控制:
    通过 WebShell 或 SSH 获得服务器权限
```

## 5. 修复方案详解

### 5.1 防御层一：页面名称白名单

**原理**：限制可访问的页面名称为预定义的集合，不在白名单中的请求直接拒绝。

```python
# ✅ 允许的页面名称白名单
ALLOWED_PAGES = {'help', 'about', 'faq', 'contact'}

# 白名单检查（第一道防线）
if name not in ALLOWED_PAGES:
    print(f"[页面加载-拦截] 非白名单页面请求: {name}")
    page_content = "<h1>页面不存在</h1>"
    return render_template("index.html", page_content=page_content)
```

**为什么白名单是最有效的防御？**

```
白名单原理:
  ┌──────────────────────────────────────┐
  │                                      │
  │  用户输入: "help"                     │
  │  检查: "help" in {"help","about",...} │
  │  结果: ✅ 通过 → 返回帮助页面          │
  │                                      │
  │  用户输入: "../../../etc/passwd"       │
  │  检查: "../../../etc/passwd" in {...} │
  │  结果: ❌ 拒绝 → 页面不存在            │
  │                                      │
  │  用户输入: "../app.py"                │
  │  检查: "../app.py" in {...}           │
  │  结果: ❌ 拒绝 → 页面不存在            │
  │                                      │
  └──────────────────────────────────────┘
  
  白名单无需理解攻击者的各种绕过手段，
  只需要知道哪些是合法的。
```

### 5.2 防御层二：真实路径规范化

**原理**：使用 `os.path.realpath()` 解析所有符号链接和 `../` 序列，得到文件系统上的真实绝对路径。

```python
import os

# 假设 PAGE_DIR = "/opt/pages"
# 用户输入 name = "../../../etc/shadow"

# ❌ 未规范化前
raw_path = os.path.join(PAGE_DIR, name)
# → "/opt/pages/../../../etc/shadow"

# ✅ 规范化后
safe_path = os.path.realpath(raw_path)
# → "/etc/shadow"
```

### 5.3 防御层三：目录边界检查

**原理**：验证规范化后的路径是否以 `pages/` 目录路径开头，确保文件在允许范围内。

```python
page_dir_real = os.path.realpath(PAGE_DIR)  # → "/opt/pages"
safe_path = os.path.realpath(os.path.join(PAGE_DIR, name))

# 检查 safe_path 是否以 page_dir_real 开头
if not safe_path.startswith(page_dir_real + os.sep):
    # ❌ 路径逃逸！文件不在 pages/ 目录内
    return "页面不存在"

# ✅ safe_path = "/etc/shadow"
# ✅ page_dir_real = "/opt/pages"
# ❌ "/etc/shadow".startswith("/opt/pages/") → False → 拦截！
```

### 5.4 防御层四：文件扩展名限制

**原理**：只允许加载 `.html` 文件，防止读取源码、数据库等非 HTML 文件。

```python
# 强制只加载 .html 文件
if not safe_path.endswith('.html'):
    safe_path += '.html'

# 之后检查文件是否存在
if os.path.isfile(safe_path):
    with open(safe_path, "r", encoding="utf-8") as f:
        page_content = f.read()
```

### 5.5 防御层五：异常捕获与安全提示

```python
try:
    with open(safe_path, "r", encoding="utf-8") as f:
        page_content = f.read()
except Exception as e:
    print(f"[页面加载] 读取失败: {e}")
    page_content = "<h1>页面不存在</h1><p>抱歉，读取页面时出错。</p>"
```

### 5.6 修复前后代码对比

| 维度 | ❌ 修复前（危险） | ✅ 修复后（安全） |
|:----:|:-----------------|:-----------------|
| **页面限制** | 无限制，任何名称都可访问 | `ALLOWED_PAGES` 白名单限制 |
| **路径构建** | `os.path.join(PAGE_DIR, name)` 直接拼接 | `os.path.realpath()` 规范化路径 |
| **目录边界** | 无检查 | `startswith(page_dir_real + os.sep)` |
| **文件类型** | 任意后缀名（自动尝试加 `.html`） | 仅 `.html` 文件 |
| **异常处理** | 无，直接崩溃 | `try/except` 安全提示 |
| **攻击日志** | 无记录 | 拦截尝试记录日志 |

```
│
├── 🛡️ 第四层：文件扩展名限制
│   ├── 只允许 .html 文件
│   └── 防止读取源码和数据库
│
└── 🛡️ 第五层：异常处理
    ├── try/except 捕获读取错误
    └── 统一的"页面不存在"返回
```
