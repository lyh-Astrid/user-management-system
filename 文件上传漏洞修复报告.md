## 1. 漏洞概述

### 1.1 什么是文件上传漏洞

文件上传漏洞（Unrestricted File Upload Vulnerability）是指 Web 应用程序在接收用户上传文件时，**未对上传文件进行充分的安全检查**，导致攻击者上传恶意文件（如 WebShell、恶意脚本、HTML 钓鱼页面等）到服务器，进而获取服务器控制权或实施进一步攻击。

### 1.2 文件上传漏洞的危害

| 危害类型 | 说明 | 严重程度 |
|:--------:|------|:--------:|
| 🔴 **WebShell 获取服务器权限** | 上传 PHP/JSP/ASP 脚本，远程执行系统命令 | **灾难性** |
| 🔴 **XSS 跨站脚本攻击** | 上传 HTML/JS 文件，在用户浏览器中执行恶意脚本 | **严重** |
| 🔴 **恶意文件分发** | 利用服务器带宽托管木马、病毒等恶意软件 | **严重** |
| 🔴 **路径遍历** | 通过 `../` 将文件写入系统关键目录 | **严重** |
| 🟡 **文件覆盖** | 同名文件互相覆盖，导致数据丢失 | **中危** |
| 🟡 **磁盘耗尽** | 上传海量或超大文件占满磁盘空间 | **中危** |

### 1.3 本次发现漏洞总览

## 2. 文件上传漏洞原理

### 2.1 安全的文件上传 vs 不安全的文件上传

```
┌────────────────────────────────────────────────────────────────────┐
│                                                                    │
│  ❌ 不安全的文件上传（修复前）：                                    │
│                                                                    │
│  用户输入 →  接收文件  →  直接保存  →  通过 /static/ 直接访问      │
│                  ↓                                                  │
│          ❌ 无扩展名检查                                            │
│          ❌ 无内容类型检查                                          │
│          ❌ 无文件名过滤                                            │
│          ❌ 无文件覆盖保护                                          │
│                                                                    │
│  ✅ 安全的文件上传（修复后）：                                      │
│                                                                    │
│  用户输入 →  接收文件  →  多层安全检查  →  安全存储  →  授权访问    │
│                            ↓                                       │
│                   ① 扩展名白名单检查（只允许图片）                    │
│                   ② 文件内容魔数检测（是否是真实图片）                │
│                   ③ 文件名安全过滤（secure_filename）               │
│                   ④ 添加时间戳防止覆盖                              │
│                   ⑤ 专用路由提供服务（非直接静态访问）               │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
```

### 2.2 文件上传攻击链路

```
攻击者
  │
  ├─→ 上传 shell.php          → 服务器存储 → 访问 http://server/shell.php → 获取控制权
  │
  ├─→ 上传 malicious.html     → 服务器存储 → 诱骗其他用户访问 → XSS 攻击
  │
  ├─→ 上传 ../../etc/cron.py  → 写入系统目录 → 定时任务执行 → 持久化控制
  │
  └─→ 上传大量大文件           → 磁盘耗尽 → 拒绝服务
```

## 3. 漏洞详情与复现

### 3.1 漏洞一：无文件类型限制（任意文件上传）

#### 漏洞信息

| 项目 | 内容 |
|------|------|
| **漏洞编号** | UPLOAD-01 |
| **位置** | `app.py` 第 134-142 行（修复前） |
| **风险等级** | 🔴 严重 |
| **CVSS 3.1** | 9.8 (Critical) |

#### 漏洞代码（修复前）

```python
@app.route("/upload", methods=["GET", "POST"])
def upload():
    ...
    if request.method == "POST":
        file = request.files.get("file")
        if file and file.filename:
            filename = file.filename                       # ❌ 直接使用原始文件名
            filepath = os.path.join(UPLOAD_FOLDER, filename)
            file.save(filepath)                             # ❌ 直接保存，无任何检查
            file_url = url_for("static", filename=f"uploads/{filename}")
```

**漏洞分析**：代码直接接收任意文件并保存到服务器，**没有对文件扩展名做任何限制**。攻击者可以上传 .php、.py、.html、.exe 等任意类型的文件。

#### 攻击复现一：上传 PHP WebShell

**攻击目标**：上传 PHP 脚本到服务器，获取远程命令执行权限。

```bash
# 攻击 payload
$ echo '<?php system($_GET["cmd"]); ?>' > shell.php

# 发送请求
$ curl -X POST http://localhost:5000/upload \
  -F "file=@/tmp/shell.php"

# 结果：文件被成功保存到 static/uploads/shell.php
```

**攻击效果**：攻击者上传了 PHP WebShell 后，可以通过以下方式在服务器上执行任意命令（如果服务器支持 PHP 解析）：

```
访问 http://server/static/uploads/shell.php?cmd=id
  → 执行系统命令 id
  → 返回服务器用户信息

访问 http://server/static/uploads/shell.php?cmd=cat+/etc/passwd
  → 读取服务器密码文件
```

#### 攻击复现二：上传 HTML 钓鱼页面（XSS）

**攻击目标**：上传 HTML 文件实现跨站脚本攻击。

```bash
# 攻击 payload
$ echo '<script>alert("XSS");</script>' > malicious.html

# 发送请求
$ curl -X POST http://localhost:5000/upload \
  -F "file=@/tmp/malicious.html"

# 结果：malicious.html 被成功保存到 static/uploads/
```

**攻击效果**：攻击者可以构造钓鱼页面，诱导其他用户访问，实现 Cookie 窃取、页面仿冒等攻击。

#### 攻击复现三：上传 Python 恶意脚本

```bash
$ echo 'import os; os.system("rm -rf /")' > evil.py
$ curl -X POST http://localhost:5000/upload -F "file=@/tmp/evil.py"
# 结果：evil.py 被成功保存到服务器
```

### 3.3 漏洞三：路径遍历攻击

#### 漏洞信息

| 项目 | 内容 |
|------|------|
| **漏洞编号** | UPLOAD-03 |
| **位置** | `app.py` 第 138 行 `os.path.join`（修复前） |
| **风险等级** | 🔴 严重 |

#### 漏洞分析

攻击者在文件名中插入 `../` 路径跳转符，可以将文件写入**上传目录之外**的系统目录。

```

  os.path.join("static/uploads/", "avatar.png")
  → static/uploads/avatar.png      ✅ 在预期目录内

路径遍历攻击:
  filename = "../../etc/cron.py"
  os.path.join("static/uploads/", "../../etc/cron.py")
  → static/uploads/../../etc/cron.py
  → etc/cron.py                    ❌ 写入系统目录！
```

#### 攻击复现

```bash
# 攻击 payload：文件名包含 ../ 路径跳转
$ curl -X POST http://localhost:5000/upload \
  -F "file=@/tmp/evil_script.png;filename=../../etc/cron.py"

# 攻击效果：文件被写入 /etc/cron.py（系统定时任务目录）
```

## 4. 修复方案详解

### 4.1 修复一：文件扩展名白名单

#### 修复代码

```python
# ✅ 允许上传的文件扩展名白名单
ALLOWED_EXTENSIONS = {'png', 'jpg', 'jpeg', 'gif', 'webp', 'bmp', 'svg'}

def allowed_file(filename):
    """检查文件扩展名是否在白名单中"""
    return '.' in filename and filename.rsplit('.', 1)[1].lower() in ALLOWED_EXTENSIONS
```

#### 修复原理

```
文件名: shell.php
        ↓
shell.php.rsplit('.', 1) → ['shell', 'php']
        ↓
'php'.lower() in ALLOWED_EXTENSIONS
        ↓
'php' not in {'png', 'jpg', 'jpeg', 'gif', 'webp', 'bmp', 'svg'}
        ↓
❌ 拒绝上传！

文件名: avatar.png
        ↓
'png' in ALLOWED_EXTENSIONS
        ↓
✅ 允许上传！
```

#### 防御效果

| 上传文件 | 扩展名 | 是否允许 | 结果 |
|:--------:|:------:|:--------:|:----:|
| `shell.php` | php | ❌ 不在白名单 | 🚫 拦截 |
| `evil.py` | py | ❌ 不在白名单 | 🚫 拦截 |
| `malicious.html` | html | ❌ 不在白名单 | 🚫 拦截 |
| `avatar.png` | png | ✅ 在白名单 | ✅ 允许 |
| `photo.jpg` | jpg | ✅ 在白名单 | ✅ 允许 |

### 4.2 修复二：文件内容类型检测

#### 修复代码

```python
# ✅ 使用 imghdr 检测文件内容是否为真实图片
file.seek(0)
file_head = file.read(512)          # 读取文件头（魔数）
file.seek(0)

image_type = imghdr.what(None, file_head)  # 检测文件内容类型
if image_type is None:
    print(f"[上传-拦截] 用户 {username} 上传的文件内容不是合法图片: {filename}")
    return render_template("upload.html", error="文件内容不是合法的图片格式！", username=username)
```

#### 修复原理

每种图片格式的文件头部都有固定的**魔数（Magic Number）**，`imghdr` 通过读取文件头来判断文件是否是真实的图片：

```
文件格式    文件头（十六进制）              魔数字符串
─────────  ─────────────────────────  ────────────────
PNG        89 50 4E 47 0D 0A 1A 0A   \x89PNG\r\n\x1a\n
JPEG       FF D8 FF E0                \xff\xd8\xff\xe0
GIF        47 49 46 38 39 61           GIF89a
BMP        42 4D                        BM

PHP 代码:  3C 3F 70 68 70               <?php
            ↑ 没有对应图片魔数
            → imghdr.what() 返回 None
            → 拦截！
```

#### 防御效果

| 上传文件 | 扩展名 | 内容检测 | 结果 |
|:--------:|:------:|:--------:|:----:|
| `shell.php.jpg` | ✅ jpg 通过 | ❌ 内容不是图片（`imghdr` 返回 `None`） | 🚫 **拦截** |
| `avatar.png`（内容是图片） | ✅ png 通过 | ✅ `imghdr` 识别为 `png` | ✅ 允许 |
| `photo.jpg`（内容是图片） | ✅ jpg 通过 | ✅ `imghdr` 识别为 `jpeg` | ✅ 允许 |

### 4.3 修复三：安全文件名处理

#### 修复代码

```python
from werkzeug.utils import secure_filename

# ✅ 使用 secure_filename 清理文件名
safe_filename = secure_filename(filename)
if not safe_filename:
    return render_template("upload.html", error="文件名不合法，请重命名后上传！", username=username)
```

#### 修复原理

`secure_filename()` 函数会**过滤掉文件名中的不安全字符**：

| 原始文件名 | `secure_filename()` 处理结果 | 说明 |
|:----------:|:---------------------------:|:----:|
| `avatar.png` | `avatar.png` | 正常文件名，保持不变 |
| `../../etc/passwd.png` | `etc_passwd.png` | `../` 被移除，`/` 被替换为 `_` |
| `shell.php` | `shell.php` | 仅做安全处理，不含路径 |
| `../../../etc/cron.py` | `etc_cron.py` | 路径遍历被彻底清除 |
| `<script>.png` | `script_.png` | 特殊字符被移除 |
| `../../../etc/hack` | `etc_hack` | 路径跳转完全失效 |

> **注意**：`secure_filename` 处理的是**路径安全**，它不能替代扩展名白名单。两者需配合使用。

### 4.4 修复四：唯一文件名防覆盖

#### 修复代码

```python
import time

# ✅ 添加时间戳前缀防止文件覆盖
timestamp = str(int(time.time()))
unique_filename = f"{timestamp}_{safe_filename}"
filepath = os.path.join(UPLOAD_FOLDER, unique_filename)
```

#### 修复原理

```
用户 A 上传 avatar.png:
  timestamp = 1783590000
  unique_filename = "1783590000_avatar.png"
  保存为: uploads/1783590000_avatar.png

用户 B 上传 avatar.png:
  timestamp = 1783590123  (不同时间)
  unique_filename = "1783590123_avatar.png"
  保存为: uploads/1783590123_avatar.png

→ 两个文件互不覆盖！✅
```

### 4.5 修复五：专用文件服务路由

#### 修复代码

```python
# ✅ 通过专用路由安全地提供上传文件
file_url = url_for("uploaded_file", filename=unique_filename)

@app.route("/uploads/<filename>")
def uploaded_file(filename):
    """安全地提供上传的文件"""
    safe_filename = secure_filename(filename)  # 二次安全校验
    if not safe_filename:
        abort(404)
    return send_from_directory(UPLOAD_FOLDER_STATIC, safe_filename)
```

#### 修复效果

| 对比项 | ❌ 修复前 (`/static/uploads/`) | ✅ 修复后 (`/uploads/`) |
|:------:|:-----------------------------:|:-----------------------:|
| 访问方式 | 直接静态文件访问 | 专用路由 + `send_from_directory` |
| 可执行脚本 | 服务器可能解析 PHP 等脚本 | 以二进制流方式提供，不会执行 |
| 路径遍历 | 可能访问目录外文件 | `secure_filename` 二次过滤 |
| 文件名验证 | 无验证 | 有 |

| `app.py` | 配置 | `UPLOAD_FOLDER = static/uploads/` | `UPLOAD_FOLDER = uploads/`（移出 static）新增 `ALLOWED_EXTENSIONS` 白名单 |
| `app.py` | 上传函数 | 直接 `file.save(filepath)` | 4 道安全检测：扩展名 → 内容 → 文件名 → 防覆盖 |
| `app.py` | 文件访问 | `url_for("static", ...)` | 专用路由 `@app.route("/uploads/<filename>")` |
| `templates/upload.html` | 注释 | `不检查文件类型和后缀名` | `已添加安全防护：后缀名白名单、内容类型检测、文件名过滤` |

#### 完整修复后的上传流程

```
用户 POST /upload
  │
  ├─① 是否已登录？
  │   └─ 否 → 302 重定向到 /login
  │
  ├─② 是否有文件且文件名非空？
  │   └─ 否 → 返回错误提示
  │
  ├─③ 文件扩展名是否在白名单中？
  │   └─ 否 → 拦截，返回"不允许上传该类型的文件"
  │
  ├─④ secure_filename 清理文件名
  │   └─ 结果为空 → 拦截，返回"文件名不合法"
  │
  ├─⑤ imghdr 检测文件内容是否为真实图片
  │   └─ 不是图片 → 拦截，返回"文件内容不是合法的图片格式"
  │
  ├─⑥ 添加时间戳生成唯一文件名
  │
  ├─⑦ 保存文件到 uploads/ + static/uploads/
  │
  └─⑧ 返回图片 URL（通过专用路由 /uploads/<filename>）
```

## 6. 安全加固总结

### 6.1 安全文件上传的黄金法则

```
┌──────────────────────────────────────────────────────────┐
│               🥇 安全文件上传黄金法则                      │
│                                                          │
│   1. 扩展名白名单（只允许需要的类型）                       │
│   2. 文件内容检测（验证魔数，不信任 Content-Type）          │
│   3. 文件名安全处理（secure_filename 防路径遍历）           │
│   4. 唯一文件名（时间戳/UUID 防覆盖）                      │
│   5. 文件大小限制（MAX_CONTENT_LENGTH）                   │
│   6. 专用文件服务路由（非直接静态访问）                     │
│   7. 上传目录不可执行（禁止脚本解析）                       │
│                                                          │
│      "Trust the content, not the extension"               │
│        信任内容本身，而不是扩展名                           │
└──────────────────────────────────────────────────────────┘
```

### 6.2 已实施的安全防护层级

```
文件上传安全防护
├── 🔴 前置拦截
│   ├── ✅ 登录认证（未登录用户不可上传）
│   └── ✅ 文件大小限制（MAX_CONTENT_LENGTH=16MB）
│
├── 🟡 文件验证层
│   ├── ✅ 扩展名白名单（仅7种图片格式）
│   ├── ✅ 文件内容魔数检测（imghdr）
│   └── ✅ 文件名安全过滤（secure_filename）
│
└── 🟢 存储与访问层
    ├── ✅ 唯一文件名（时间戳防覆盖）
    ├── ✅ 专用路由提供访问（非静态目录直出）
    └── ✅ 访问时二次安全校验
```

### 6.3 防御绕过方式对比

| 绕过方式 | 防御措施 | 是否有效 |
|:---------|:---------|:--------:|

| 修改 Content-Type 为 `image/jpeg` | 内容魔数检测（不信任请求头） | ✅ 防御 |
