## 1. SSRF 漏洞概述

### 1.1 什么是 SSRF

**服务端请求伪造（Server-Side Request Forgery, SSRF）** 是一种攻击方式，攻击者利用服务器端应用程序的功能，**诱使服务器向攻击者指定的地址发起请求**。由于请求是从服务器内部发起的，攻击者可以访问本应被防火墙隔离的内部系统。

```
┌───────────────────────────────────────────────────────────────┐
│                                                               │
│  SSRF 攻击三要素:                                              │
│                                                               │
│  ① 应用程序提供 URL 抓取/请求功能                               │
│  ② 对用户提交的 URL 不做充分校验                                │
│  ③ 服务器可以访问内部网络资源                                   │
│                                                               │
│  SSRF 的核心危害：                                             │
│  利用服务器"内部人员"身份访问本应不可达的资源                     │
│                                                               │
│  外部攻击者 ──不能直接访问──→ 内网服务器                        │
│  外部攻击者 ──利用 SSRF ──→ 目标服务器 ──访问──→ 内网服务器     │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

### 1.2 SSRF 与 CSRF 的区别

| 对比维度 | SSRF（服务端请求伪造） | CSRF（跨站请求伪造） |
|:---------|:----------------------|:--------------------|
| **受害方** | 服务器本身 | 已登录的用户 |
| **请求发起者** | 服务器 | 用户浏览器 |
| **访问目标** | 内网系统/本地文件 | 外网 Web 应用 |
| **核心危害** | 内网渗透、数据泄露 | 以用户身份执行操作 |

## 2. SSRF 攻击原理

### 2.1 SSRF 攻击链路

```
攻击者
  │
  ├─→ POST /fetch-url    url=file:///etc/passwd
  │       ↓
  │   服务器读取本地文件   ← 利用 file:// 协议绕过 Web 限制
  │       ↓
  │   返回文件内容给攻击者
  │
  ├─→ POST /fetch-url    url=http://127.0.0.1:5000/admin
  │       ↓
  │   服务器向内网发起请求  ← 利用服务器内部网络权限
  │       ↓
  │   返回内网页面给攻击者
  │
  └─→ POST /fetch-url    url=http://192.168.1.1:8080
          ↓
      服务器扫描内网端口  ← 利用服务器做跳板
          ↓
      返回端口开放信息
```

### 2.2 漏洞代码分析

**漏洞代码（修复前）**：

```python
@app.route("/fetch-url", methods=["POST"])
def fetch_url():
    url = request.form.get("url", "")

    # ❌ 直接访问，无任何校验
    req = urllib.request.Request(url)
    resp = urllib.request.urlopen(req, timeout=10)
    content = resp.read()
```

**漏洞根本原因**：

| 缺失的防护 | 说明 |
|:-----------|:------|
| ❌ 无协议限制 | file:// 可读取本地文件，dict:// 可与内网服务交互 |
| ❌ 无内网 IP 过滤 | 可访问 127.0.0.1、10.x.x.x、192.168.x.x 等 |
| ❌ 无主机名检查 | localhost 可解析到 127.0.0.1 |
| ❌ 无 DNS 解析校验 | 无法区分域名指向内网还是外网 |
| ❌ 无超时限制 | `timeout=10` 已设，但无连接数限制 |

## 3. 漏洞详情与复现

### 3.1 SSRF-01：file:// 协议读取本地文件

#### 3.1.1 漏洞信息

| 项目 | 内容 |
|:-----|:------|
| **漏洞编号** | SSRF-01 ~ SSRF-04 |
| **攻击向量** | `file://` 协议 |
| **风险等级** | 🔴 灾难性 |
| **CVSS 3.1** | 9.8 (Critical) |

#### 3.1.2 攻击复现

```bash
# 攻击请求：读取系统密码文件
POST /fetch-url
url=file:///etc/passwd

📦 返回内容：
  root:x:0:0:root:/root:/bin/bash
  daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
  ...

# 攻击请求：读取密码哈希
POST /fetch-url
url=file:///etc/shadow

📦 返回内容：
  root:$y$j9T$...hash...:19609:0:99999:7:::
  ...

# 攻击请求：读取应用源码
POST /fetch-url
url=file:///opt/app.py

📦 返回 Flask 应用完整源码，包括：
  - secret_key = "dev-key-2025"
  - 数据库路径和结构
  - 所有业务逻辑代码

# 攻击请求：读取用户数据库
POST /fetch-url
url=file:///opt/data/users.db

📦 返回 SQLite 数据库文件，包含所有用户数据
```

#### 3.1.3 危害分析

```
file:// 协议允许攻击者读取服务器文件系统上的任意文件：

  /etc/passwd     → 系统用户列表       → SSH 爆破目标
  /etc/shadow     → 密码哈希          → 离线破解
  /opt/app.py     → 应用源码          → 发现更多漏洞
  /opt/*.db       → 数据库文件        → 所有用户数据
  /root/.ssh/*    → SSH 私钥          → 服务器登录
  /etc/nginx/*    → 服务器配置        → 攻破其他站点
```

### 3.3 SSRF-03：内网 IP 扫描与端口探测

#### 3.3.1 漏洞信息

| 项目 | 内容 |
|:-----|:------|
| **漏洞编号** | SSRF-07 ~ SSRF-09 |
| **攻击向量** | 私有 IP 地址段扫描 |
| **风险等级** | 🔴 严重 |

#### 3.3.2 内网 IP 范围

| 范围 | 地址段 | 掩码 |
|:-----|:-------|:----:|
| A 类私有地址 | 10.0.0.0/8 | 255.0.0.0 |
| B 类私有地址 | 172.16.0.0/12 | 255.240.0.0 |
| C 类私有地址 | 192.168.0.0/16 | 255.255.0.0 |
| 回环地址 | 127.0.0.0/8 | 255.0.0.0 |

### 3.4 SSRF-04：DNS Rebinding 攻击风险

DNS Rebinding 是一种高级 SSRF 绕过技术。攻击者注册一个域名，初始解析指向外网 IP（通过白名单检查），然后快速切换解析到内网 IP。由于 DNS 缓存的时间差，服务器会误认为请求目标仍然是外网地址。

### 3.5 SSRF-05：其他危险协议

| 协议 | 功能 | 危害 |
|:-----|:------|:------|
| `file://` | 读取本地文件 | 🔴 任意文件读取 |
| `dict://` | 与内网服务交互 | 🟡 探测内网 Redis/Memcached |
| `gopher://` | 发送任意 TCP 数据 | 🔴 利用内网 Redis 反弹 Shell |
| `ftp://` | FTP 协议 | 🟡 内网 FTP 探测 |

## 5. 修复方案详解

### 5.1 防御层一：协议白名单

**原理**：只允许 http:// 和 https:// 协议，拒绝 file://、dict://、gopher:// 等危险协议。

```python
# ✅ 只允许 http/https 协议
if not url.lower().startswith(('http://', 'https://')):
    print(f"[SSRF-拦截] 非 http/https 协议: {url}")
    return render_template("index.html", fetch_error="仅支持 http:// 和 https:// 协议的 URL！")
```

**防御效果**：

| 协议 | 修复前 | 修复后 |
|:-----|:------:|:------:|
| `http://example.com` | ✅ 正常 | ✅ 正常 |
| `https://example.com` | ✅ 正常 | ✅ 正常 |
| `file:///etc/passwd` | ✅ 读取成功 | ❌ 拦截 |
| `file:///etc/shadow` | ✅ 读取成功 | ❌ 拦截 |
| `dict://127.0.0.1:6379` | ✅ 可能交互 | ❌ 拦截 |
| `gopher://...` | ✅ 可能交互 | ❌ 拦截 |

### 5.2 防御层二：主机名黑名单

**原理**：直接屏蔽 localhost、127.0.0.1、0.0.0.0 等回环地址。

```python
# ✅ 禁止访问内网/回环地址的关键词
blocked_hosts = {'localhost', '127.0.0.1', '0.0.0.0', '[::1]', '::1'}
if hostname and hostname.lower() in blocked_hosts:
    print(f"[SSRF-拦截] 内网地址: {url}")
    return render_template("index.html", fetch_error="不允许访问该地址！")
```

### 5.3 防御层三：DNS 解析与 IP 范围检查

**原理**：将域名解析为 IP 地址，然后检查是否属于私有 IP 范围。这可以防御域名指向内网的情况。

```python
import socket

# ✅ 解析 IP 地址，检查是否为内网 IP
ip = socket.gethostbyname(hostname)

# 检查私有 IP 范围
if ip.startswith('10.') or ip.startswith('192.168.') or ip == '127.0.0.1':
    raise ValueError("内网地址")

# 检查 172.16.x.x ~ 172.31.x.x
if ip.startswith('172.'):
    parts = ip.split('.')
    if len(parts) == 4 and 16 <= int(parts[1]) <= 31:
        raise ValueError("内网地址")
```

**私有 IP 范围检测逻辑**：

```
用户提交 URL: http://internal.company.com/admin
  ① 协议检查: http:// → ✅ 通过
  ② 主机名检查: "internal.company.com" → ✅ 不在黑名单
  ③ DNS 解析: "internal.company.com" → "10.0.1.5" (内网IP)
  ④ IP 范围检查: "10.0.1.5".startswith("10.") → ❌ 拦截！
```

### 5.4 修复前后代码对比

| 维度 | ❌ 修复前 | ✅ 修复后 |
|:----:|:---------|:----------|
| **协议限制** | 无限制 | 仅 http:// 和 https:// |
| **主机名黑名单** | 无 | localhost, 127.0.0.1, 0.0.0.0, ::1 |
| **DNS 解析** | 无 | `socket.gethostbyname()` 解析域名 |

| **超时控制** | 10 秒 | 10 秒（保持） |

### 5.5 完整防御流程图

```
用户提交 URL
  │
  ├─❶ 协议检查: 是否 http/https?
  │   └─ 否 → ❌ 拦截 "仅支持 http/https 协议"

  ├─❷ 主机名黑名单: 是否 localhost/127.0.0.1/0.0.0.0?
  │   └─ 是 → ❌ 拦截 "不允许访问该地址"

  │   └─ 解析失败 → ⚠️ 记录日志，继续（可能允许）
  │
  ├─❹ 私有 IP 检测:
  │   ├─ 10.x.x.x?       → ❌ 拦截
  │   ├─ 192.168.x.x?    → ❌ 拦截
  │   ├─ 172.16-31.x.x?  → ❌ 拦截
  │   └─ 127.x.x.x?      → ❌ 拦截
  │
  └─❺ 通过 → ✅ 执行抓取
```

## 7. 安全加固总结

### 7.1 三层 SSRF 防御架构

```
SSRF 防御架构
├── 🛡️ 第一层：协议白名单
│   └── 仅允许 http:// 和 https://
│
├── 🛡️ 第二层：主机名黑名单
│   └── 禁止 localhost/127.0.0.1/0.0.0.0
│
└── 🛡️ 第三层：IP 范围检测
    ├── DNS 解析域名
    ├── 10.x.x.x       → 拦截
    ├── 192.168.x.x    → 拦截
    ├── 172.16-31.x.x  → 拦截
    └── 127.x.x.x      → 拦截
```

### 7.2 安全 URL 抓取黄金法则

```
┌──────────────────────────────────────────────────────────┐
│             🥇 安全 URL 抓取黄金法则                       │
│                                                          │
│   1. 协议白名单：只允许 http/https                         │
│   2. 内网 IP 过滤：阻止所有私有地址                        │
│   3. DNS 解析检查：域名可能指向内网                        │
```
